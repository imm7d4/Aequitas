# US-12.2 - User & Wallet Administration (Maker-Checker)

**Epic:** Admin & Audit Controls  
**Phase:** Phase 12 - Admin & Audit  
**Status:** Defined (Refined)  
**Priority:** High  
**Complexity:** Medium  

## User Story

As a **Support Representative**,  
I want to manage user identities and perform wallet interventions using a **Maker-Checker** workflow,  
so that financial adjustments are audited, scoped by **JIT grants**, and approved by a second party.

## Acceptance Criteria

### 1. User Identity & KYC Compliance
- [ ] **Role-Based KYC**: Only Compliance or PlatformAdmin roles can update KYC statuses.
- [ ] **Strict State Machine**: Transitions: `PENDING` → `VERIFIED/REJECTED`; `VERIFIED` → `SUSPENDED`.
- [ ] **Data Privacy**: Mask IP addresses (e.g., `192.168.x.x`) for general Support; Unmasking requires AuditAdmin approval + Justification.

### 2. Wallet Interventions (Maker-Checker & JIT Scoping)
- [ ] **Maker-Checker Workflow**:
  - Support acts as **Maker** (stages the adjustment).
  - PlatformAdmin acts as **Checker** (approves after Step-Up MFA).
- [ ] **JIT Scoped Grants**:
  - Access granted for specific `max_amount`, `max_uses`, and `expires_at`.
  - Manual adjustment power auto-revokes upon limit hit or expiry.
- [ ] **Dual Approval**: Adjustments > ₹50,000 require approval from **two independent Checkers**.

### 3. Ledger Integrity & Auditing
- [ ] **Idempotency**: Every adjustment requires a unique `adjustment_id` and mandatory `Reason Code`.
- [ ] **Validation**: Backend must verify `balance == sum(credits) - sum(debits)` before and after any change.

## Business Analyst Insights
> [!CAUTION]
> Manual wallet adjustments represent the highest financial risk. Implementing JIT scoping and Dual Approval mitigates both accidental errors and malicious insider threats.
