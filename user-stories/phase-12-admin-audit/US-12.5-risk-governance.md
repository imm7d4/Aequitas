# US-12.5 - Risk Governance & ABAC

**Epic:** Admin & Audit Controls  
**Phase:** Phase 12 - Admin & Audit  
**Status:** Defined (Refined)  
**Priority:** High  
**Complexity:** Medium  

## User Story

As a **Risk Officer**,  
I want to define risk parameters using a **Hybrid RBAC/ABAC** model (Role + Action + Context),  
so that I can enforce safety thresholds that adapt to login context and JIT status.

## Acceptance Criteria

### 1. Risk-Based Access Control
- [ ] **Contextual Restriction**: Restrict sensitive modifications if the admin is logged in from a non-whitelisted IP or unusual hours.
- [ ] **Hybrid Model**: Permissions = `Role (Risk Officer)` + `Action (Update Margin)` + `Context (JIT Active & Amount < Limit)`.

### 2. Margin & Threshold Management
- [ ] **Global Overrides**: Ability to adjust global margin multipliers and circuit breaker sensitivities (requires Checker approval).
- [ ] **Anomalous Pattern Monitoring**: View reports on "Suspicious Trading Clusters" and "Insider Threat" scores for admin accounts.

### 3. Fail-Safe Operations
- [ ] **Logging Mandate**: If Audit Ingestion fails, the system **MUST BLOCK** all administrative "Write" actions.
- [ ] **JIT Expiry Handling**: Automatically cancel any pending parameter requests if the requester's JIT window expires.

## Business Analyst Insights
> [!IMPORTANT]
> Risk governance must be dynamic. By moving to a Hybrid ABAC model, we ensure that administrative power is only exercised within the safe, approved context of an active, scoped JIT window.
